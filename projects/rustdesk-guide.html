RustDesk Self-Hosting — Homelab

Self-hosted RustDesk ID and Relay servers on Linux or Docker. Provides secure remote access to internal PCs and servers without relying on third-party cloud infrastructure. Ideal for homelab or small business use.

1. Prerequisites

  • Linux host (Ubuntu/Debian recommended)
  • Docker + Docker Compose (optional but recommended)
  • Firewall capable of forwarding TCP 21114–21119
  • Public DNS record, e.g. rustdesk.mydomain.com

RustDesk uses 21114–21119/TCP for ID, NAT traversal, relay, and console.

2. Deploy RustDesk with Docker Compose

Create the RustDesk directory: 


mkdir -p /opt/rustdesk
cd /opt/rustdesk

Create docker-compose.yml:


version: "3.8"

services:
  hbbs:
    image: rustdesk/rustdesk-server
    container_name: rustdesk-hbbs
    command: hbbs -r hbbr:21117
    restart: unless-stopped
    ports:
      - "21114:21114"
      - "21115:21115"
      - "21116:21116"
      - "21117:21117"
    volumes:
      - ./data:/root/.config/rustdesk-server

  hbbr:
    image: rustdesk/rustdesk-server
    container_name: rustdesk-hbbr
    command: hbbr
    restart: unless-stopped
    ports:
      - "21118:21118"
      - "21119:21119"
      

docker compose up -d
docker compose ps

Both containers should show Up.

3. Firewall / Reverse Proxy Publishing

For direct WAN access, create inbound NAT rules:

  • 21114/TCP — ID Registration
  • 21115/TCP — NAT Traversal
  • 21116/TCP — Web Console
  • 21117–21119/TCP — Relay

If using Cloudflare Tunnel or HAProxy, forward raw TCP to the RustDesk server.

4. Configure RustDesk Clients

  1. Open RustDesk → Settings → Network
  2. ID Server: rustdesk.yourdomain.com:21114
  3. Relay Server: rustdesk.yourdomain.com:21117
  4. Restart RustDesk
  5. Confirm status shows Ready

5. Hardening & Best Practices

  • Limit WAN access to trusted IP ranges
  • Use Cloudflare Tunnel for protected TCP access
  • Backup the ./data directory
  • Rotate container images regularly
  • Enable strong unattended access passwords

6. Troubleshooting

  • “Not Ready” → DNS or port forwarding not correct
  • Internal works, external fails → missing NAT rules
  • Random disconnects → ISP NAT or IDS interference
  • Container errors → check docker logs hbbs and hbbr
