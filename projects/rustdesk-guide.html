RustDesk Self-Hosting for Homelab

A self-hosted RustDesk ID and Relay server stack running on Linux or Docker. Provides secure remote access to internal PCs and servers without relying on third-party cloud systems.

1. Prerequisites

  • Ubuntu/Debian Linux host or VM
  • Docker + Docker Compose installed (recommended)
  • Firewall/Router able to forward TCP ports 21114–21119
  • Public DNS record such as rustdesk.mydomain.com

RustDesk uses 21114–21119/TCP for ID, NAT traversal, relay, and console.

2. Deploy RustDesk with Docker Compose

Create the RustDesk directory:

mkdir -p /opt/rustdesk
cd /opt/rustdesk

Create docker-compose.yml: 

version: "3.8"

services:
  hbbs:
    image: rustdesk/rustdesk-server
    container_name: rustdesk-hbbs
    command: hbbs -r hbbr:21117
    restart: unless-stopped
    ports:
      - "21114:21114"
      - "21115:21115"
      - "21116:21116"
      - "21117:21117"
    volumes:
      - ./data:/root/.config/rustdesk-server

  hbbr:
    image: rustdesk/rustdesk-server
    container_name: rustdesk-hbbr
    command: hbbr
    restart: unless-stopped
    ports:
      - "21118:21118"
      - "21119:21119"
docker compose up -d
docker compose ps

Both containers should show Up.

3. Firewall / Reverse Proxy Publishing

Forward these ports on your router/firewall:

  • 21114/TCP — ID Registration
  • 21115/TCP — NAT Traversal
  • 21116/TCP — Web Console
  • 21117–21119/TCP — Relay + backend

For Cloudflare Tunnel or HAProxy, forward raw TCP to the server.

4. Configure RustDesk Clients

  1. Open RustDesk → Settings → Network
  2. ID Server: rustdesk.yourdomain.com:21114
  3. Relay Server: rustdesk.yourdomain.com:21117
  4. Restart RustDesk
  5. Confirm status shows Ready

5. Hardening & Best Practices

  • Restrict inbound RustDesk ports to trusted IP ranges
  • Use Cloudflare Tunnel for encrypted TCP proxying
  • Backup the ./data directory regularly
  • Update container images on every RustDesk release
  • Use strong unattended access passwords

6. Troubleshooting

  • “Not Ready” → DNS/port forwarding incorrect
  • Works internal, fails external → missing WAN NAT
  • Disconnects → ISP NAT or IDS/IPS interference
  • Container errors → check docker logs hbbs & hbbr
